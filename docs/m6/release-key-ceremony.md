# Release-Key Ceremony — v1.0.0 MANIFEST Signing

M6 Phase 0a (L-60). One-way decision locked 2026-04-23: software Ed25519 with passphrase-encrypted storage (zero-cost). Hardware upgrade (YubiKey) available as v1.0.1 editorial errata later.

**Day-2 hard deadline.** Everything else in M6 depends on this key existing and being reachable at Phase 2e.

## Why the discipline matters

Once MANIFEST.json.jws ships with a signature over the v1.0.0 artifact set, the public key fingerprint is a **permanent fixture** of the spec. Rotating the key after release requires a v1.0.1 errata with explicit migration language. Losing the key pre-release costs days of restart. Leaking the key means every downstream verification is compromised.

Execute the steps below in order. Do not skip. Do not improvise.

## Step 1 — Pick the signing machine

Use a machine you trust and intend to keep accessible through at least v1.0.1. The key lives on this machine encrypted; it comes out of encryption only during the ceremony and at release-day signing.

- Full-disk encryption enabled
- OS patched
- No shared logins
- Password-protected screensaver
- Backup of the disk recent

Do NOT use: shared dev VM, ephemeral cloud instance, machine used for development with untrusted dependencies, CI runner.

## Step 2 — Generate the Ed25519 keypair

```bash
# Creates soa-release-v1.0.key (private) and soa-release-v1.0.pub (public)
openssl genpkey -algorithm ed25519 -out soa-release-v1.0.key
openssl pkey -in soa-release-v1.0.key -pubout -out soa-release-v1.0.pub
```

Compute the public-key fingerprint (this goes into MANIFEST.json as `publisher_kid`):

```bash
openssl pkey -pubin -in soa-release-v1.0.pub -outform DER 2>/dev/null | \
    openssl dgst -sha256 -binary | \
    openssl base64
```

Record this fingerprint. It must match what lands in the signed MANIFEST.

## Step 3 — Wrap the private key with a passphrase

```bash
# Encrypts in place; prompts for passphrase twice
openssl pkey -in soa-release-v1.0.key -aes-256-cbc -out soa-release-v1.0.key.enc
rm soa-release-v1.0.key  # destroy unencrypted copy
```

**Passphrase requirements:**
- Generated by a password manager (not invented)
- ≥ 24 characters
- Never reused
- Never typed into a browser
- Never committed anywhere

## Step 4 — Back it up

You need **two** independent copies. One failure mode: single-machine disk failure. Another: passphrase forgotten. Backup covers the first; passphrase recovery is on you.

**Copy 1 — offline, geographically separate:**
- Encrypted USB stick
- Stored in a location not co-located with the signing machine
- Labeled `soa-release-v1.0 — DO NOT INSERT UNLESS RELEASING`

**Copy 2 — encrypted cloud vault (e.g., 1Password secure note):**
- Upload `soa-release-v1.0.key.enc` as an attachment
- Store the passphrase separately in the same vault (different item)
- Do NOT put both the encrypted key and the passphrase in the same file

Verify backup integrity before moving on:

```bash
# Decrypt a copy from Copy 1 to /tmp and verify it matches
openssl pkey -in /tmp/test-copy.key.enc -text -noout
# You should see "ED25519 Private-Key" — if not, re-copy.
rm /tmp/test-copy.key.enc
```

## Step 5 — Public key goes into the repo

```bash
cp soa-release-v1.0.pub "C:/Users/wbrumbalow/Documents/Projects/soa-harness=specification/keys/soa-release-v1.0.pub"
```

Commit:
```bash
git add keys/soa-release-v1.0.pub
git commit -m "M6 Phase 0a: add release public key (Ed25519, kid=<fingerprint>)"
```

`keys/soa-release-v1.0.pub` is public-facing. Anyone verifying a signed MANIFEST fetches it from this repo.

## Step 6 — Signing-machine hygiene checklist

Before every real signing operation:
- [ ] Machine is the designated signing machine (no improvisation)
- [ ] No untrusted browser tabs open
- [ ] No video calls running
- [ ] No screen-share running
- [ ] Fresh terminal session (not a sudo'd or persistent session)
- [ ] Passphrase pulled from password manager, not typed from memory
- [ ] Private key decrypted to a `tmpfs`/RAM disk where available, never to `/tmp` on disk
- [ ] Decrypted key file wiped (`shred -u`) immediately after signing

## Step 7 — Dry-run signing

```bash
# Test-sign a dummy payload to confirm the workflow
echo '{"test":"dry-run"}' | \
    openssl pkeyutl -sign -inkey soa-release-v1.0.key -rawin > dry-run.sig

# Verify with the public key
openssl pkeyutl -verify -pubin -inkey soa-release-v1.0.pub -rawin -in <(echo '{"test":"dry-run"}') -sigfile dry-run.sig
```

If verify reports "Signature Verified Successfully", you're ready for Phase 2e. Destroy dry-run artifacts.

## Disaster scenarios

- **Private key leaked before release**: destroy the key, regenerate, restart ceremony. No MANIFEST has been signed yet so nothing downstream is compromised.
- **Private key leaked after v1.0.0 release**: v1.0.1 errata with rotated key. Document the leak and rotation in the errata. All v1.0.0 signatures become suspect and must be re-verified via out-of-band channel.
- **Passphrase forgotten**: unrecoverable. Restart ceremony from Step 2. Public key file in repo becomes abandoned; new one lands via v1.0.1 errata.
- **Signing machine destroyed, backup intact**: decrypt backup on a new trusted machine, continue.
- **Both signing machine AND backups destroyed**: restart ceremony. Ship v1.0.1 with new key.
