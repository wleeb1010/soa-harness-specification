# v1.0.x Long-Term Support Branch Policy

M7 prerequisite (L-61). Defines what adopters of v1.0.0 can count on while M7-M11 develop v1.1+.

## Branches

- **`main`** — active development. All M7+ work lands here. Version: v1.1.0 and above.
- **`v1.0-lts`** — cut from `v1.0.0` tag (commit landing v1.0.0 on all three repos). Bug-fix and security-patch only. Version: v1.0.1, v1.0.2, ... through v1.0.Z.

## Who maintains what

- `main`: solo maintainer (full-time during M7-M11)
- `v1.0-lts`: solo maintainer (as-needed; SLA below)

## What goes to `v1.0-lts`

✅ **YES — lands on `v1.0-lts` as a patch release:**
- Critical security vulnerabilities (CVEs, sandbox escapes, auth bypass, credential leaks in logs/audit)
- Data-loss bugs (session corruption, audit-chain breakage)
- Compliance-blocking doc errors (e.g., spec section claiming A when impl does B)
- License metadata corrections (SPDX errors, LICENSE file typos)

❌ **NO — lands on `main` only, adopters on v1.0.x must wait for v1.1.0+:**
- New features
- Performance improvements (unless fixing a data-loss race)
- Refactors
- New test IDs
- Additional adapters
- Documentation improvements beyond correctness fixes

## Service-Level Agreements

| Severity | Classification | Response time | Patch release |
|---|---|---|---|
| **CRITICAL** | Security (RCE, auth bypass, data exfil) OR data loss | **≤48 hours** acknowledge + initial diagnosis | **≤5 business days** v1.0.Z published |
| **HIGH** | Compliance-blocking (spec-impl divergence) OR severe UX broken | ≤7 calendar days acknowledge | ≤14 calendar days v1.0.Z published |
| **MEDIUM** | Minor doc or metadata correctness | ≤30 calendar days acknowledge | Next scheduled v1.0.Z |

These SLAs are best-effort commitments from a single maintainer. Pilots in the adopter-concierge program get priority escalation.

## Reporting a bug against v1.0.x

1. GitHub issue on the relevant repo, labeled `v1.0-lts`
2. For security issues, use GitHub Security Advisories (private)
3. Include: version installed (`npm ls @soa-harness/runner`), platform, minimal repro

## How a v1.0.x patch ships

Per `docs/errata-policy.md` editorial tier:

1. Fix lands on `v1.0-lts` branch (not `main`)
2. If the same fix is also needed on `main`, cherry-pick forward
3. Bump spec_version editorial in affected sibling repos per the 1.0.N cadence
4. Regenerate + re-sign MANIFEST.json.jws per `docs/m6/release-key-ceremony.md`
5. Publish npm packages at 1.0.N using the same release-day runbook as v1.0.0
6. `gh release create v1.0.N` on the affected repos
7. Append entry to `ERRATA.md`

Rollback playbook: `docs/m6/rollback-runbook.md` applies identically to v1.0.x patches.

## v1.0-lts end-of-life

The `v1.0-lts` branch is maintained for **at least 12 months from the v1.0.0 release date** (2026-04-23 → 2027-04-23 minimum). After that:

- **Option A (default):** archive the branch. Final v1.0.Z tag documented. No further fixes.
- **Option B:** if a pilot adopter or Bake-Off partner is still on v1.0.x in good faith, extend for an additional 6 months.
- **Option C:** if v2.0.0 breaks v1.0-lts conformance irrecoverably, maintain v1.0-lts in perpetuity until all known adopters have migrated.

Decision made 12 months after release, not committed to now.

## Automation

CI workflow `.github/workflows/v1.0-lts-patches.yml` in each repo watches the `v1.0-lts` branch for commits and:

- Runs the full test suite against the branch head
- Rejects merges if `v1.0-lts` ever would contain features newer than spec_version 1.0
- Alerts the maintainer via email + GitHub notifications on CRITICAL-labeled issues
